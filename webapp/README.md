# webapp/

The browser-side surface of r2-compiler. Currently **Phase 2-preview** — a static catalogue browser. The eventual **Phase 2-full** is a WASM R2 hive with the visual canvas, source viewer, build/author/deploy event flow.

## Status (2026-05-31): Phase 2-preview — static catalogue browser

What's here right now:
- A single static HTML page listing every catalogue entry, with click-to-inspect detail + file viewer.
- No WASM hive yet, no R2 events, no compile triggering.
- Built specifically to make the catalogue layout **visible early**, so design intent can be reviewed before deep commits to the orchestrator + WASM-hive architecture.

## Running it

```bash
# From the repo root:
./webapp/run.sh
# Opens http://localhost:8080/webapp/index.html
```

The script:
1. Runs `python3 tools/build-catalogue-index.py` to refresh `webapp/dist/manifest.json` from the on-disk catalogue.
2. Starts `python3 -m http.server 8080` rooted at the repo (so the page can fetch `../catalogue/...` files relative to `webapp/index.html`).

Re-run `run.sh` after authoring a new catalogue entry to pick up the change. (Phase 2-full will do this live via filesystem watching.)

## What you can do with it

- **Browse boards** (left panel) — see the three carriers, their target triples, flash size, compulsory plugins, full pinout, etc.
- **Browse ensembles** (right panel) — see `rocker-sensor`, expand to its nested plugins (`sensor/lis2dh`) and sentants (`Identity`).
- **Click an entry** to open its detail view: structured-artefact metadata + a file tree + a file viewer.
- **Click any file** (board.toml, BOARD.md, AI-CONTEXT.md, lis2dh's src/lib.rs, the Identity sentant.yaml, …) to read its content.
- **Header pill** shows the (placeholder) project context. Project switching is captured in memory as a Phase 2-full feature — see [[project-projects-concept]].

## Files

```
webapp/
├── index.html              # the single page
├── styles/main.css         # dev-tool aesthetic — monospace, restrained palette
├── ui/app.js               # ~250 lines vanilla JS — fetches manifest, renders, file viewer
├── dist/manifest.json      # generated by tools/build-catalogue-index.py
├── run.sh                  # convenience launcher (rebuild + serve)
└── README.md               # this file
```

Zero JS dependencies. No npm, no build step beyond the Python manifest generator. Designed to deploy to GitHub Pages as-is for a public catalogue browser once we want one (small `gh-pages` workflow runs the Python builder + commits the result).

## Phase 2-full notes (what comes next)

When this preview is replaced by the real WASM-hive webapp:
- `ui/app.js` becomes the JS glue around the WASM bundle (per the r2-workshop `webapp/` pattern using `crates/r2-wasm`).
- `webapp/crate/` (currently empty) holds the Rust crate compiled to `wasm32-unknown-unknown` that runs the R2 hive (Catalogue / Composition / SourceViewer / Builder sentants).
- The orchestrator binary (`orchestrator/`, currently empty) serves the WASM bundle from `localhost:21050` and hosts the workstation R2 hive.
- `manifest.json` is replaced by a live `r2.compiler.catalogue.*` event stream the WASM hive consumes.
- The project switcher becomes a working pane backed by `r2.compiler.project.*` events.

The static preview is throwaway — Phase 2-full is a clean rewrite from r2-workshop's `webapp/` as the template, not an evolution of this preview.
